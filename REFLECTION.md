# Reflection

## Архитектурные решения

**MapClient как абстракция** — провайдер карт вынесен за интерфейс (`MapClient`), реализован через `TwoGisClient`. Смена провайдера — замена одного класса в `get_map_client()` без изменений в остальном коде. Это позволяет легко переключиться на OpenStreetMap, Яндекс.Карты или любой другой источник тайлов и геоданных без рефакторинга бизнес-логики.

**Скоринг без LLM** — оценка мест (шум, розетки, рейтинг) реализована детерминированно в `tools/score_places.py`, а не через LLM-вызов. Это делает результат стабильным, воспроизводимым и не тратит токены на простые эвристики. LLM задействован только там, где реально нужна семантика: разбор запроса пользователя и формирование финального текстового ответа.

**Pydantic-модели как контракт** — `SearchRequest`, `Place`, `ScoredPlace` фиксируют формат между инструментами и агентом. Это упрощает отладку: каждый слой (поиск → обогащение → скоринг → ответ) можно тестировать независимо, передавая сериализованные объекты. Pydantic также даёт бесплатную валидацию входных данных от API — если 2GIS вернёт неожиданный формат, ошибка будет явной и локализованной, а не молчаливой.

**PydanticAI как агентный фреймворк** — выбор в пользу PydanticAI вместо LangChain или голого OpenAI SDK обусловлен типобезопасностью: инструменты декларируются с Pydantic-схемами, и LLM получает структурированный интерфейс. Это снижает вероятность галлюцинаций при вызове инструментов.

**Кэш запросов** — результаты поиска и обогащения сохраняются в `data/cache_*.json`. При повторном запросе с теми же параметрами API не вызывается. Это важно при разработке: демо-ключ 2GIS ограничен 1000 запросов в месяц, и без кэша квота быстро исчерпывается в ходе итераций.

**FastAPI + Leaflet.js** — серверная часть минимальна: один эндпоинт `/search`, один `/map`. Вся интерактивность карты сосредоточена в `frontend/app.js` на Leaflet.js. Такое разделение позволяет заменить фронтенд (например, на React) без изменений в Python-коде.
